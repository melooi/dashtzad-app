# Dashtzad Memory — مغز مرکزی پروژه

این skill حافظهٔ دائمی و نقشهٔ ذهنیِ پروژهٔ دشت‌زاد است. هدفش سه چیز است:

1. **دیگر کل پروژه خوانده نشود.** برای جهت‌گیری فقط `PROJECT-MAP.md` (کوچک) + یک memory فایلِ مرتبط خوانده می‌شود، نه کل کدبیس. این مستقیماً مصرف توکن را پایین می‌آورد.
2. **هیچ کاری گم نشود.** هر تغییر کد در memory و log ثبت می‌شود؛ تصمیم‌ها، فایل‌های جدید، gotchaها و وضعیت واقعی همان‌جا می‌ماند.
3. **حافظه با هر کار قوی‌تر شود.** هر task باید memory مرتبط را بهتر از قبل ترک کند (حداقل یک واقعیت/تصمیم/درس جدید اضافه کند). مثل یک ذهن انسانی که با هر تجربه دقیق‌تر می‌شود.

ساختار فایل‌ها (همه داخل پوشهٔ همین skill):
```
dashtzad-memory/
  SKILL.md            ← همین فایل (قوانین اجباری)
  PROJECT-MAP.md      ← نقشهٔ مرکزی / مایندمپ (همیشه اول خوانده می‌شود؛ کوچک بماند)
  memory/
    00-architecture.md   ← استک قفل‌شده، کانونشن‌ها، چیزهای مرده/آرشیو
    01-admin-panel.md
    02-storefront.md
    03-chat-ai.md
    04-content-cms.md
    05-commerce.md
    06-integrations.md
    07-design-system.md
    08-data-model.md
  logs/
    CHANGELOG.md      ← لاگ append-only، تاریخ‌دار
```

## حلقهٔ اجباری (هر task، بدون استثنا)

پیش از نوشتن هر کد، این ۷ قدم را دقیقاً اجرا کن:

1. **MAP اول.** `PROJECT-MAP.md` را بخوان. این تنها راهِ مجازِ جهت‌گیری است. هرگز برای فهمیدن «کجای کاریم» کل `src/` را scan نکن.
2. **دامنه را تشخیص بده.** task به کدام دامنه(ها) مربوط است؟ (admin / storefront / chat-ai / content-cms / commerce / integrations / design-system / data-model).
3. **فقط memory همان دامنه را بخوان.** `memory/NN-domain.md`. اگر task واقعاً چنددامنه‌ای است، حداکثر ۲ تا را بخوان — نه بیشتر.
4. **کار را انجام بده.** طبق تصمیم‌های قفل‌شدهٔ ثبت‌شده در آن memory و در `00-architecture.md`.
5. **تأیید کن (اجباری قبل از ادعای «done»).** `npx tsc --noEmit` و در صورت لزوم `npm run build`. نتیجهٔ واقعی را یادداشت کن. **هرگز** «done» نگو بدون اجرای واقعی این‌ها.
6. **لاگ بزن.** یک entry تاریخ‌دار به `logs/CHANGELOG.md` اضافه کن (قالب پایین). و memory دامنه را به‌روز کن: فایل‌های جدید، تصمیم، gotcha، وضعیت، تاریخِ آخرین تأیید.
7. **MAP را به‌روز کن.** خط وضعیت همان دامنه را در `PROJECT-MAP.md` تازه کن. اگر دامنهٔ جدیدی ساخته شد، به map و memory اضافه‌اش کن.

اگر هر کدام از این قدم‌ها را رد کردی، کار ناقص است.

## قوانین سخت (نقض = خطا)

- **هرگز برای جهت‌گیری کل پروژه را نخوان.** map + یک memory. اگر اطلاعات لازم در memory نبود، فقط همان فایل‌های مشخصِ مرتبط را باز کن، نه کل درخت.
- **هرگز وضعیت جعلی نساز.** اگر تأیید نشده، بنویس «تأییدنشده». «احتمالاً درست است» = تأییدنشده.
- **هرگز «done» بدون شواهد.** tsc/build واقعی، با نتیجهٔ واقعی ثبت‌شده.
- **تصمیم‌های قفل‌شده قفل‌اند.** در `00-architecture.md` لیست‌اند. هیچ‌کدام بدون تأیید صریح کاربر باز نمی‌شوند. تعارض را flag کن، خودسرانه عوض نکن.
- **داده/کد جعلی نساز.** اگر بک‌اند یا داده نیست، empty/pending state حرفه‌ای؛ نه mock پنهان.
- **هر task حافظه را قوی‌تر می‌کند.** حداقل یک نکتهٔ جدید (تصمیم، gotcha، نگاشت فایل) در memory مرتبط ثبت کن.
- **memory را کوچک و واقعی نگه‌دار.** اگر چیزی کهنه، تکراری، یا غلط شد، **بازسازی‌اش کن** (پاک کن/ادغام کن/اصلاح کن). map هیچ‌وقت نباید از ۱ صفحهٔ قابل‌اسکن بزرگ‌تر شود.
- **پنل ادمین و storefront زبان طراحی واحد دارند** (`dz-` tokens، `dz-night` برای دارک). جزئیات در `07-design-system.md`.

## Bootstrap (فقط بار اول، یا وقتی کاربر گفت «بازسازی کن»)

اگر memory فایل‌ها خالی/ناقص‌اند، یک بار این کار را بکن — حافظه را از روی **کد واقعی** بساز، نه از حدس:

1. `git log --oneline -20` و `git status --short` بگیر تا وضعیت واقعی commitها را بدانی.
2. `npx tsc --noEmit` بزن و خطاهای موجود را به‌عنوان «بدهی» در `08-data-model.md` یا دامنهٔ مربوط ثبت کن.
3. ساختار `src/` را در سطح بالا scan کن (فقط نام پوشه‌ها/فایل‌های کلیدی، نه محتوای کامل همه).
4. هر memory فایل را با قالب پایین پر کن: هدف، وضعیت واقعی، تصمیم‌های قفل‌شده، فایل‌های کلیدی، threadهای باز، gotchaها.
5. `PROJECT-MAP.md` را با خط وضعیت هر دامنه + بدهی‌های شناخته‌شده کامل کن.
6. در `logs/CHANGELOG.md` یک entry «BOOTSTRAP» بزن.

بعد از bootstrap، دیگر هیچ‌وقت کل پروژه برای جهت‌گیری خوانده نمی‌شود.

```markdown
## هدف
این دامنه چه کاری می‌کند (۲–۳ جمله).

## تصمیم‌های قفل‌شده
- ...

## فایل‌های کلیدی
- `path` — نقش

## وضعیت فعلی (واقعی، نه آرزویی)
- ✅ <چیزی که تأیید شده>
- 🟡 <نیمه‌کاره>
- ⏳ <شروع‌نشده>
- ⚠️ <بدهی/مشکل>

## threadهای باز
- <کار ناتمامی که باید ادامه یابد>

## Gotchas (درس‌های گران)

```

## قالب entry لاگ (`logs/CHANGELOG.md`، append-only، جدیدترین بالا)

```markdown
## <تاریخ ISO> — <دامنه> — <عنوان کوتاه>
- چه شد: ...

- وضعیت بعد: ✅/🟡/⏳/⚠️
- نکتهٔ جدید برای memory: ...
```

## نکتهٔ توکن (چرا این skill مهم است)

بیشترین مصرف توکنِ این پروژه از session‌های با context بالای ۱۵۰k و خواندنِ مکرر کل کدبیس بوده. این skill دقیقاً برای حل همان است: **map + یک memory** معمولاً چند هزار توکن است، نه چند صد هزار. کارهای اجرایی را روی Sonnet ببر؛ این skill را همان‌جا هم کامل می‌توان اجرا کرد.
